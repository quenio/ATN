# Model Context Protocol Design

This document refines [`MCP-2025-11-25.spec.md`](./MCP-2025-11-25.spec.md) into an implementable ATN design.

## Design Drivers

- Preserve the specified host-client-server session architecture.
- Allocate responsibility for JSON-RPC lifecycle correctness, capability-gated features, task handling, and authorization binding.
- Keep transport, authorization, and feature subsystems separable so implementations can vary without violating the specification.
- Make control points explicit for user consent, schema validation, subscriptions, pagination, and task state transitions.

## Assumptions

- A concrete implementation may realize one host, one or more clients, and one or more servers as processes, libraries, or services, but the design treats them as components with explicit interfaces.
- Transport adapters realize stdio and Streamable HTTP specifics while exposing a common message channel to the protocol core.
- Capability declaration and negotiation are modeled as registries consulted by dispatch and validation components rather than as ad hoc checks spread across handlers.
- Authorization details defined by external OAuth and discovery standards are represented through dedicated components that produce server-bound access tokens and validated HTTP session state.
- User consent is enforced by host-side control components before outbound tool invocation authorization and sampling authorization records are created.

## Notes

- The design centers on a session coordinator and protocol core, with feature handlers separated by MCP capability area so downstream implementation can map them to modules, services, or packages.
- Host-side consent gates are explicit because the specification makes user authorization structural rather than advisory for tool calls and sampling.
- Authorization handling is separated from transport adaptation so HTTP token carriage, metadata discovery, PKCE coordination, and audience validation can evolve independently while still preserving server binding.
- Task handling is isolated behind a task registry, task state store, and transition policy so task augmentation can be added to tools, sampling, and elicitation without weakening lifecycle constraints.

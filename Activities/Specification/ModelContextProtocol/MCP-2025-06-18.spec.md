# Model Context Protocol 2025-06-18 Requirements Specification

This document converts the natural-language requirements stated directly in `Activities/Specification/ModelContextProtocol/MCP-2025-06-18.md` into an ATN specification.

## Extracted Requirements

### Protocol structure and defined capabilities

- MCP uses JSON-RPC 2.0 messages.
- MCP uses stateful connections.
- MCP performs server capability negotiation.
- MCP performs client capability negotiation.
- MCP distinguishes hosts, clients, and servers.
- Hosts are LLM applications that initiate connections.
- Clients are connectors within host applications.
- Servers are services that provide context and capabilities.
- MCP defines server feature categories for resources, prompts, and tools.
- MCP permits client feature categories for sampling, roots, and elicitation.
- MCP includes utilities for configuration, progress tracking, cancellation, error reporting, and logging.
- The protocol intentionally limits server visibility into prompts.

### Mandatory security requirements

- Users must explicitly consent to all data access and operations.
- Users must understand all data access and operations.
- Users must retain control over what data is shared.
- Users must retain control over what actions are taken.
- Hosts must obtain explicit user consent before exposing user data to servers.
- Hosts must not transmit resource data elsewhere without user consent.
- Tools must be treated with appropriate caution because they represent arbitrary data access and code execution paths.
- Hosts must obtain explicit user consent before invoking any tool.
- Users must explicitly approve any LLM sampling requests.

### Recommended requirements and guidance

- Implementors should provide clear UIs for reviewing and authorizing activities.
- User data should be protected with appropriate access controls.
- Tool descriptions, such as annotations, should be considered untrusted unless obtained from a trusted server.
- Users should understand what each tool does before authorizing its use.
- Users should control whether sampling occurs at all.
- Users should control the actual prompt that will be sent during sampling.
- Users should control what sampling results the server can see.
- Implementors should build robust consent and authorization flows into their applications.
- Implementors should provide clear documentation of security implications.
- Implementors should implement appropriate access controls and data protections.
- Implementors should follow security best practices in their integrations.
- Implementors should consider privacy implications in their feature designs.

## Assumptions

- Because the source file is a top-level overview page rather than the full linked specification, this ATN model captures only the requirements stated directly in that file.
- The ATN specification models one representative protocol definition (`mcp`) and one representative implementation (`implementation`) to express protocol-wide and implementation-wide properties compactly.
- Server features and client features are modeled as feature categories defined by the protocol, not as features that every individual server or client must implement.
- SHOULD-level guidance is preserved as recommendations in the ATN model and is not elevated into mandatory conformance constraints.

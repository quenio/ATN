# Model Context Protocol 2025-06-18 Requirements Specification

This document converts the natural-language requirements stated in `Activities/Specification/ModelContextProtocol/MCP-2025-06-18.md` into an ATN specification.

## Extracted Requirements

### Core protocol architecture

- MCP uses JSON-RPC 2.0 messages over stateful connections.
- Hosts are LLM applications that initiate connections.
- Clients are connectors within host applications.
- Servers are services that provide context and capabilities.
- Hosts coordinate multiple clients, enforce security policies, handle user authorization decisions, and maintain security boundaries between servers.
- Clients establish one stateful session per server, negotiate capabilities, route messages bidirectionally, and maintain isolation between servers.
- Servers expose resources, prompts, and tools; clients may expose sampling, roots, and elicitation.
- Implementations must support the base protocol and lifecycle management; other protocol components are optional.
- The protocol is designed so servers do not see the whole conversation or other servers' state unless the host deliberately shares context.

### Security principles and baseline trust requirements

- Users must explicitly consent to and understand data access and operations.
- Users must retain control over shared data and taken actions.
- Hosts must obtain explicit user consent before exposing user data to servers.
- Hosts must not transmit resource data elsewhere without user consent.
- Tools must be treated with caution, and tool descriptions or annotations should be treated as untrusted unless they come from trusted servers.
- Hosts must obtain explicit user consent before invoking tools.
- Users must explicitly approve sampling requests and should control whether sampling occurs, the prompt that will be sent, and what results the server can see.
- Implementors should provide clear review and authorization UIs, robust consent flows, access controls, documentation, and privacy-aware designs.

### Base messaging, schema, metadata, and icons

- Requests must have string or integer IDs, IDs must not be null, and request IDs must be unique within a session.
- Result responses must reuse the request ID and include a `result` field.
- Error responses must reuse the request ID when available, include an `error` object with integer `code` and `message`, and notifications must receive no response and must not include IDs.
- Schemas without `$schema` default to JSON Schema 2020-12.
- Clients and servers must support JSON Schema 2020-12, validate against the declared or default dialect, and gracefully handle unsupported dialects.
- Schemas must be valid under their effective dialect.
- Implementations must not assume values at reserved `_meta` keys.
- Clients that render icons must support PNG and JPEG, should support SVG and WebP, and must reject unsafe icon URIs.
- Icon consumers must treat icon metadata as untrusted, avoid credentialed fetching, verify same-origin use, and validate MIME types and file contents before rendering.

### Authorization and OAuth-based security requirements

- Authorization is optional overall, but HTTP-based transports should follow the authorization specification, while stdio transports should instead use environment credentials.
- Authorization servers must implement OAuth 2.1 with appropriate security measures.
- MCP servers must implement OAuth 2.0 Protected Resource Metadata, and MCP clients must use it for authorization server discovery.
- Authorization servers must provide at least one discovery mechanism from OAuth Authorization Server Metadata or OpenID Connect Discovery, and MCP clients must support both.
- MCP servers must publish authorization server locations and discovery metadata, and clients must prefer `resource_metadata` from `WWW-Authenticate` before falling back to well-known URIs.
- Clients must parse `WWW-Authenticate` headers and treat challenged scopes as authoritative for the current request.
- Clients should follow the specified registration priority order across pre-registration, Client ID Metadata Documents, dynamic registration, and manual entry.
- Client ID Metadata Documents must use HTTPS URLs with required fields, and authorization servers must validate fetched documents, redirect URIs, and document structure.
- Clients must request least privilege, use the OAuth `resource` parameter in authorization and token requests, and use the target MCP server's canonical URI.
- Access tokens must be sent in `Authorization` headers on every HTTP request and must not appear in query strings.
- Servers must validate tokens and intended audience, reject invalid or expired tokens with HTTP 401, and accept only tokens valid for their own resources.
- Insufficient-scope failures should use HTTP 403 with `WWW-Authenticate`, and clients should use step-up authorization with retry limits.
- Implementations must follow OAuth security best practices, use HTTPS for authorization server endpoints, restrict redirect URIs to HTTPS or localhost, and implement secure token storage.
- Clients must implement PKCE, use `S256` when capable, verify advertised PKCE support before proceeding, and refuse authorization if required metadata is absent.
- Authorization servers must validate exact redirect URIs, prevent open redirection, and expose `code_challenge_methods_supported` when using OpenID Connect discovery for MCP compatibility.
- Proxy servers must obtain per-client user consent before forwarding to third-party authorization systems and must not pass through client-presented upstream tokens.

### Security best practices beyond core OAuth flows

- Proxy servers must store consent per client, secure consent cookies, validate redirect URIs exactly, and create and validate secure single-use OAuth state values only after consent is approved.
- Server-side MCP clients must mitigate SSRF when fetching OAuth-related URLs, should require HTTPS in production, should block private and reserved IP ranges, and should validate redirect targets.
- Servers implementing authorization must verify every inbound request and must not use sessions as authentication.
- Session identifiers must be secure and unpredictable and should be bound to user-specific information.
- Clients that support one-click local MCP server configuration must obtain explicit consent before executing commands, must show the exact command, and should sandbox local servers.
- Servers intended for local execution should prefer stdio or otherwise restrict local HTTP access.

### Lifecycle, transports, cancellation, ping, and progress

- Initialization must be the first interaction.
- Clients must send `initialize` with protocol version, capabilities, and client implementation information.
- Servers must respond with protocol version, capabilities, and server implementation information.
- Clients must send `notifications/initialized` after successful initialization.
- Before initialization completes, clients and servers should limit outbound requests to the specific exceptions defined by the protocol.
- Version negotiation must use supported protocol versions, should prefer the latest supported versions, and clients should disconnect if the returned version is unsupported.
- During operation, both sides must respect the negotiated version and negotiated capabilities.
- Implementations should establish timeouts, cancel timed-out requests, allow configurable timeouts, and enforce maximum timeouts even when progress arrives.
- JSON-RPC messages must be UTF-8 encoded.
- Clients should support stdio whenever possible.
- In stdio, messages are newline-delimited, must not contain embedded newlines, and stdin/stdout traffic must contain only valid MCP messages.
- Streamable HTTP servers must expose one MCP endpoint supporting POST and GET, validate `Origin`, return HTTP 403 on invalid origin, and should bind locally to localhost and authenticate connections.
- HTTP clients must POST one JSON-RPC message per request, send the required `Accept` headers, and support either JSON or SSE responses to requests.
- SSE retry, polling, resumability, non-broadcast delivery, and same-stream replay rules must be respected.
- Session IDs must be secure, visible ASCII, propagated in `MCP-Session-Id` headers, and new sessions must be created after HTTP 404 session termination responses.
- HTTP requests after initialization must include `MCP-Protocol-Version`, and invalid or unsupported versions must receive HTTP 400.
- Custom transports must preserve JSON-RPC and lifecycle requirements and should be documented.
- Cancellation notifications may target only same-direction in-progress requests, may not cancel `initialize`, and task cancellation must use `tasks/cancel` instead.
- Ping receivers must respond promptly with an empty result.
- Progress tokens must be unique strings or integers, progress values must increase, progress must reference active requests only, and task progress must keep using the original token until terminal completion.

### Task requirements

- Tasks are experimental and require explicit capability declaration during initialization.
- Requestors should augment only request types whose task capability was declared by the receiver.
- Tool-level task support must also respect each tool's `execution.taskSupport` setting.
- Task IDs must be strings, generated by the receiver, and unique.
- Tasks must begin in `working` and may only follow the specified lifecycle transitions.
- `input_required` tasks should be used when the receiver needs requestor input and must carry related-task metadata.
- Task responses must include `createdAt`, `lastUpdatedAt`, actual `ttl`, and may include `pollInterval`.
- Accepted task requests must return `CreateTaskResult` promptly.
- `tasks/result` must either return the final underlying result immediately for terminal tasks or block until the task becomes terminal.
- Task-related messages must include `io.modelcontextprotocol/related-task` metadata except where the protocol explicitly says not to include it.
- Task status notifications are optional and requestors must not rely on them.
- Task listing must use opaque cursors, return `nextCursor` when more tasks exist, and include all tasks otherwise retrievable by `tasks/get` for that requestor.
- Cancellation of terminal tasks must fail with invalid params; once cancelled, a task must remain cancelled.
- When authorization context is available, receivers must bind tasks to it; otherwise they must use cryptographically strong task IDs and should avoid advertising `tasks.list` if requestors cannot be identified.
- Receivers should rate limit task operations, enforce resource limits, clean up expired tasks, and audit task activity.

### Elicitation, roots, and sampling

- Clients that support elicitation must declare at least one elicitation mode, and servers must not send modes the client did not declare.
- Form mode must not be used for sensitive secrets; URL mode must be used for such interactions.
- Clients must clearly identify the requesting server, protect privacy, and provide review, decline, and cancel controls.
- Elicitation requests must include a human-readable message and a mode; omitted mode defaults to form mode.
- Form mode uses a restricted flat primitive-schema subset.
- URL mode requires a valid URL and an `elicitationId`.
- Completion notifications for URL mode may be sent, but only to the initiating client and with the original `elicitationId`; clients must ignore unknown or completed IDs.
- `URLElicitationRequiredError` may only be used when URL elicitation is actually required and must list URL-mode elicitations with IDs.
- Elicitation state must be securely bound to verified users, not only to session IDs.
- URL mode must not be used to authorize users to the MCP server itself, and third-party credentials obtained via URL mode must never transit the MCP client.
- Servers and clients must follow safe URL-handling rules, including not embedding sensitive data in elicitation URLs, not prefetching them, requiring explicit user consent, and opening them in a way that does not expose their contents to the MCP client or model.
- Servers must not trust client-provided identity claims and must ensure that the same user who initiated a URL-mode elicitation is the user who completes it.
- Roots-capable clients must declare the roots capability and must emit list-changed notifications when they advertise that support.
- Root URIs must be `file://` URIs.
- Clients must expose only authorized roots, validate them, apply access controls, and monitor accessibility; servers should respect root boundaries.
- Sampling-capable clients must declare sampling support, and tool-enabled sampling requires `sampling.tools` capability.
- Servers must not send tool-enabled sampling requests to clients that did not declare `sampling.tools`.
- Context-inclusion values other than `none` are soft-deprecated and should only be used when the client declares the related capability.
- Sampling should keep a human in the loop for request and response review.
- Tool-result user messages must contain only tool results.
- Every assistant message containing tool uses must be followed by a user message containing matching tool results before any other message.
- Sampling implementations must handle sensitive data appropriately and should limit tool-loop iteration depth.

### Prompts, resources, tools, completion, logging, and pagination

- Prompt servers must declare prompt capability and should emit list-changed notifications when supported.
- Prompt image and audio content must be base64-encoded with valid MIME types, and embedded resources must include URI, MIME type, and actual text or blob content.
- Prompt inputs and outputs must be validated carefully to prevent injection or unauthorized access.
- Resource servers must declare resources capability and should emit supported notifications.
- Resource servers must validate resource URIs, properly encode binary content, and ensure custom URI schemes follow RFC 3986.
- `https://` resource URIs should be used only when the client can fetch them directly.
- Tool servers must declare tools capability and should emit list-changed notifications when supported.
- Tool input schemas must be valid non-null JSON Schema objects, and tool annotations must be treated as untrusted unless their source is trusted.
- If a tool provides an output schema, the server must return conforming structured content and clients should validate it.
- Tool servers must validate inputs, enforce access controls, rate limit invocations, and sanitize outputs.
- Clients should request confirmation for sensitive tool operations, show tool inputs before invocation, validate tool results, and use call timeouts.
- Completion servers must declare completions capability; completion results are limited to 100 values.
- Completion implementations must validate inputs, rate limit, protect sensitive suggestions, and prevent completion-based information disclosure.
- Logging servers must declare logging capability.
- Log messages must not contain credentials, secrets, personal identifying information, or internal details that would aid attacks.
- Pagination uses opaque cursors and server-determined page sizes.
- Clients must not parse, modify, or persist cursors across sessions and should treat missing `nextCursor` as the end of results.
- Servers should provide stable cursors, and invalid cursors should produce invalid-params errors.

## Assumptions

- Although the repository filename is `MCP-2025-06-18.md`, the source document itself states that it is a consolidated local copy of MCP specification pages under version `2025-11-25`; this specification follows the requirements stated in that consolidated content.
- Because ATN is relational rather than operational, sequencing constraints and protocol procedures are modeled as conformance predicates on representative protocol profiles.
- Representative implementation constants such as `protected_client`, `task_receiver`, and `tool_server` stand for any implementation conforming to that role-specific requirement set.
- MAY-level permissions are modeled explicitly only where they materially constrain valid protocol behavior; the extracted requirements section retains additional optional permissions and guidance in prose without incorrectly elevating them to mandatory conformance rules.
- Repeated requirements restated in the generated schema reference portion of the source file are captured once in the ATN model rather than duplicated.

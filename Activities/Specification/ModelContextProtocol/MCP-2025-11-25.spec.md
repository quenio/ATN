# Model Context Protocol 2025-11-25 Requirements Specification

This document translates the natural-language requirements in `Activities/Specification/ModelContextProtocol/MCP-2025-11-25.md` into an ATN specification.

## Extracted Requirements

### Architecture and base protocol

- MCP is a client-host-server protocol built on JSON-RPC 2.0 over stateful connections.
- Hosts initiate connections, manage multiple clients, enforce security and consent boundaries, and preserve isolation between servers.
- Each client is associated with one server connection and negotiates capabilities with that server.
- Servers expose resources, prompts, and tools; clients may expose roots, sampling, and elicitation.
- All implementations must support the base protocol and lifecycle management; other feature families are optional.
- Requests must use string or integer IDs that are non-null and unique per requestor within a session.
- Result and error responses reuse the request ID when available; notifications have no IDs and receive no response.
- Protocol interactions are versioned, capability-negotiated, and constrained to the negotiated version and capabilities.

### Security and authorization

- Hosts must require explicit user consent for exposing user data, invoking tools, and approving sampling.
- Servers should not gain visibility into unrelated conversation state or other servers' state.
- JSON Schema 2020-12 is the default schema dialect; clients and servers must support it and validate schemas under the effective dialect.
- Reserved `_meta` keys must not be assumed by implementations.
- Icons are untrusted inputs; clients that render icons must support PNG and JPEG, reject unsafe URIs, and validate fetched content safely.
- Authorization is optional overall, but HTTP transports should follow the OAuth-based MCP authorization model.
- Protected MCP servers must publish protected resource metadata and identify authorization servers through supported discovery mechanisms.
- Clients must use protected resource metadata, support OAuth and OpenID discovery, use the OAuth `resource` parameter, use PKCE, and send bearer tokens only in `Authorization` headers.
- Servers must validate tokens, audience binding, redirect URIs, and scopes, and must not pass through upstream tokens.
- Proxy servers must obtain per-client consent before forwarding to third-party authorization systems.

### Lifecycle, transports, and utilities

- Initialization is the first interaction: clients send `initialize`, servers reply with version/capabilities/info, and clients then send `notifications/initialized`.
- JSON-RPC messages must be UTF-8 encoded.
- Stdio messages are newline-delimited MCP messages only.
- Streamable HTTP uses a single endpoint supporting POST and GET, validates `Origin`, supports JSON and SSE responses, and uses secure session IDs when sessions are assigned.
- HTTP requests after initialization must include `MCP-Protocol-Version`, and invalid or unsupported values must be rejected.
- Cancellation applies only to same-direction in-progress requests and must not cancel `initialize`; task cancellation uses `tasks/cancel` instead.
- Ping requests must receive prompt empty results.
- Progress tokens are strings or integers, are unique across active requests, and progress notifications reference active requests only.

### Tasks, elicitation, roots, and sampling

- Tasks are experimental, capability-negotiated durable request wrappers with receiver-generated unique string IDs.
- Tasks begin in `working`, transition only through the permitted lifecycle, and expose `createdAt`, `lastUpdatedAt`, `ttl`, and optional `pollInterval`.
- `tasks/result` returns the underlying terminal result; task-related messages use related-task metadata except where the protocol defines method parameters as the source of truth.
- Task access must bind to authorization context when available; otherwise task IDs must be unguessable.
- Elicitation supports `form` and `url` modes; clients must declare supported modes and servers must not use undeclared modes.
- Sensitive information must not be requested through form elicitation and instead requires URL-mode elicitation.
- URL-mode elicitation requires a valid URL and elicitation ID, must be user-consented, and must not expose third-party credentials through the client.
- Roots-capable clients expose file-URI roots only and must validate, authorize, and monitor those roots.
- Sampling-capable clients must declare sampling support; tool-enabled sampling also requires `sampling.tools` capability.
- Sampling tool use must preserve the required tool-use/tool-result message balance, and user review remains central.

### Prompts, resources, tools, completion, logging, and pagination

- Prompt, resource, tool, completion, and logging support are all capability-gated.
- Prompt and tool content may include text, image, audio, and embedded resources; binary prompt content must be base64-encoded and MIME-typed.
- Resources are URI-addressed, can be listed/read/template-addressed, and may support subscriptions and list-changed notifications.
- Resource update notifications should follow prior subscriptions.
- Tool input schemas must be valid non-null JSON Schema objects; tool output schemas, when present, constrain structured results.
- Tool annotations must be treated as untrusted unless their source is trusted.
- Completion results are bounded and reference either prompts or resource templates.
- Log messages must not contain credentials, secrets, personal identifying information, or attack-aiding internal details.
- Pagination uses opaque cursors that clients must not parse, modify, or persist across sessions.

## Assumptions

- The source file is a consolidated local copy of the MCP specification revision `2025-11-25`, even though the repository filename retains `2025-06-18` for continuity.
- The ATN model focuses on MCP's structural ontology: sessions, messages, capabilities, transports, tasks, elicitation objects, roots, resources, prompts, tools, and related invariants.
- Normative `MUST` and `REQUIRED` statements are modeled primarily as ATN assertions. Most `SHOULD` and `MAY` guidance remains in prose unless it materially shapes protocol structure.
- The source repeats some requirements across narrative chapters and schema-reference comments; the ATN model captures each requirement family once rather than duplicating equivalent constraints.

-- JSON-RPC message invariants:
SESSION_MESSAGES_STAY_WITHIN_SESSION_PARTIES:
  all m in Message:
    (sender(m) = client_of(session_of_message(m)) and
     receiver(m) = server_of(session_of_message(m))) or
    (sender(m) = server_of(session_of_message(m)) and
     receiver(m) = client_of(session_of_message(m)))

REQUEST_IDS_ARE_UNIQUE_PER_REQUESTOR_AND_SESSION:
  all r1, r2 in Request:
    session_of_message(r1) = session_of_message(r2) and
    sender(r1) = sender(r2) and
    request_id(r1) = request_id(r2)
      => r1 = r2

RESULT_RESPONSES_REUSE_REQUEST_IDS:
  all response in ResultResponse:
    response_id(response) = request_id(response_to(response))

ERROR_RESPONSES_REUSE_REQUEST_IDS_WHEN_READABLE:
  all response in ErrorResponse:
    request_id_was_readable(response)
      => response_id(response) = request_id(response_to(response))

RESPONSES_STAY_WITHIN_THE_REQUEST_SESSION:
  all response in Response:
    session_of_message(response) = session_of_message(response_to(response))

CANCELLATIONS_TARGET_SAME_DIRECTION_NON_INITIALIZATION_REQUESTS:
  all n in Notification:
    notification_kind(n) = cancelled_kind =>
      sender(n) = sender(cancelled_request(n)) and
      session_of_message(n) = session_of_message(cancelled_request(n)) and
      not (request_kind(cancelled_request(n)) = initialize_kind)

PINGS_RECEIVE_EMPTY_RESULTS:
  all response in ResultResponse:
    request_kind(response_to(response)) = ping_kind =>
      empty_result(response)

PROGRESS_NOTIFICATIONS_REFERENCE_ACTIVE_REQUESTS_AND_TOKENS:
  all n in Notification:
    notification_kind(n) = progress_kind =>
      progress_requested(progress_for(n)) and
      progress_token_of_notification(n) = progress_token(progress_for(n))

PROGRESS_TOKENS_ARE_UNIQUE_ACROSS_ACTIVE_REQUESTS:
  all r1, r2 in Request:
    progress_requested(r1) and
    progress_requested(r2) and
    progress_token(r1) = progress_token(r2)
      => r1 = r2

    contains_tool_result(message)

           message_content(message, content)
             => some result in ToolResultContent:
                  content = result

TOOL_USE_MESSAGES_ARE_FOLLOWED_BY_MATCHING_TOOL_RESULTS:
  all message in AssistantSamplingMessage:
    contains_tool_use(message)
      => all use in ToolUseContent:
           message_content(message, use)
             => some result in ToolResultContent:
                  message_content(next_sampling_message(message), result) and
                  tool_result_for(result) = tool_use_id(use)

EMBEDDED_RESOURCES_HAVE_URI_MIME_TYPE_AND_PAYLOAD:

RESOURCE_UPDATE_NOTIFICATIONS_FOLLOW_SUBSCRIPTIONS:

      => client_subscribes_resource(client_of(session_of_message(n)), updated_resource(n))

    tool_has_output_schema(tool)
      => tool_output_schema_valid(tool) and
         tool_structured_results_conform(tool)

    notification_kind(n) = log_message_kind
      => not log_contains_sensitive_data(n)

PAGINATION_CURSORS_ARE_OPAQUE:

    cursor_is_opaque(cursor)

CLIENTS_DO_NOT_PERSIST_CURSORS_ACROSS_SESSIONS:

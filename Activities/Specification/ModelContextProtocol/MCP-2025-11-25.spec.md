# Model Context Protocol Requirements Specification

This document translates `Activities/Specification/ModelContextProtocol/MCP-2025-11-25.md` into a structural ATN specification.

## Extracted Requirements

### Architecture and core protocol

- MCP defines a host-client-server architecture.
- A host creates and manages client instances.
- Each client maintains an isolated, stateful session with one server.
- MCP communication uses JSON-RPC 2.0 messages between clients and servers.
- Requests must carry a non-null string or integer ID that is unique for the requestor within the session.
- Result responses and ordinary error responses must correspond to a request and use that request's ID.
- Notifications do not carry request IDs and must not be answered with responses.

### Lifecycle, versions, and capabilities

- Initialization is the first interaction in a session.
- The client initiates initialization by sending protocol version, client capabilities, and client implementation information.
- The server replies with protocol version, server capabilities, and server implementation information.
- After successful initialization, the client sends an `initialized` notification.
- Clients and servers negotiate optional features through declared capabilities.
- If HTTP is used, the negotiated protocol version must be carried on subsequent HTTP requests.

### Transport and session handling

- MCP supports stdio and Streamable HTTP transports.
- JSON-RPC messages must be UTF-8 encoded.
- Streamable HTTP uses a single endpoint supporting POST and GET, with SSE used for server-to-client streaming.
- Streamable HTTP servers must validate `Origin` headers.
- HTTP sessions may carry session identifiers; if used, they must be visible ASCII and handled securely.
- Clients must start a new session after receiving HTTP 404 for an expired session identifier.

### Common structures

- Schemas used by MCP default to JSON Schema 2020-12 unless an explicit dialect is declared.
- Clients and servers must support JSON Schema 2020-12 and must validate schemas according to the declared or default dialect.
- `_meta` is reserved for protocol and extension metadata.
- Icon metadata uses HTTP(S) or `data:` URIs and safe MIME handling.

### Authorization

- Authorization is optional overall, but HTTP authorization support follows the MCP authorization specification.
- Protected MCP servers publish OAuth 2.0 Protected Resource Metadata that lists at least one authorization server.
- MCP authorization servers provide OAuth Authorization Server Metadata or OpenID Connect Discovery.
- Clients discover authorization servers from protected resource metadata and must support both discovery mechanisms.
- Clients must include the OAuth `resource` parameter in authorization and token requests.
- The `resource` parameter must identify the target MCP server by canonical URI.
- Clients must send bearer access tokens in the HTTP `Authorization` header on every HTTP request.
- Servers must validate that access tokens were issued for them as audience and must reject unrelated or passed-through tokens.
- Authorization server endpoints must use HTTPS, and redirect URIs must be localhost or HTTPS.
- PKCE is mandatory for MCP clients performing authorization code flows.
- Client ID Metadata Documents, when used, must be hosted at HTTPS URLs whose `client_id` matches the document URL exactly and includes redirect URIs.

### Utilities

- Ping is a request/response health check.
- Cancellation is a notification tied to an in-flight request and must not be used to cancel `initialize` or task-augmented requests.
- Progress notifications are keyed by a progress token supplied in request metadata.
- Task augmentation introduces task IDs, task states, task listing, task cancellation, and task-result retrieval.
- Task IDs are strings generated by the receiver and are unique for that receiver.
- Tasks begin in `working` and may move only through the allowed lifecycle transitions.
- Terminal task states (`completed`, `failed`, `cancelled`) do not transition further.
- Task-related requests, responses, and notifications are associated with the underlying task.

### Client-provided features

- Roots are client-provided filesystem boundaries; root URIs are `file://` URIs.
- Clients supporting roots may notify servers when the root list changes.
- Sampling lets servers request model generations from clients.
- Tool-enabled sampling is allowed only when the client declares sampling tool support.
- Tool-use messages in sampling must be matched by tool-result messages before the conversation continues.
- Elicitation lets servers request information from users through form mode or URL mode.
- Clients declaring elicitation must support at least one elicitation mode.
- Servers must not send elicitation requests using modes the client did not declare.
- Sensitive elicitation must use URL mode, not form mode.
- URL-mode completion notifications and URL-required errors are tied to URL elicitation identifiers.

### Server-provided features

- Prompt servers expose prompts and optional prompt-list change notifications.
- Resource servers expose resources, resource templates, optional subscriptions, and optional resource-list change notifications.
- Resource update notifications are meaningful only for subscribed resources.
- Tool servers expose tools, optional tool-list change notifications, and tool schemas.
- Tool input schemas must be valid JSON Schema objects.
- Tools may declare task support as forbidden, optional, or required.
- Completion servers provide argument completions for prompt and resource-template references.
- Logging servers accept log-level configuration requests and emit log-message notifications.
- Pagination uses opaque cursors for prompts, resources, resource templates, tools, and tasks.

### Security and consent requirements represented structurally

- Hosts must obtain explicit user authorization before invoking tools.
- Hosts must obtain explicit user authorization before allowing sampling requests.
- Clients and servers must handle sensitive data appropriately.
- MCP proxy and authorization flows must preserve client-specific consent and token audience binding.
- Session identifiers must not replace authorization.

## Assumptions

- The ATN model focuses on protocol objects, relationships, and mandatory invariants, with one or more `Session` instances representing the running protocol.
- External standards referenced by MCP—especially JSON-RPC 2.0, OAuth 2.1, RFC 9728, RFC 8414, OpenID Connect Discovery, and JSON Schema dialect semantics—are represented abstractly through MCP-facing types and relations rather than restated in full.
- Advisory `SHOULD` guidance is kept primarily in prose unless it defines reusable protocol structure or clarifies capability boundaries.
- Optional fields are modeled through specialized subtypes and auxiliary objects rather than a single generic optional-value construction.
- Pagination, metadata-key syntax, MIME safety, and other validation-oriented details are modeled where they affect structure, but not every low-level lexical rule is re-expressed as an ATN assertion.

-- Core protocol invariants:
MESSAGE_PEERS_MATCH_SESSION:
  all m in JsonRpcMessage:
    (sender(m) = session_client(session(m)) or sender(m) = session_server(session(m))) and (receiver(m) = session_client(session(m)) or receiver(m) = session_server(session(m))) and not (sender(m) = receiver(m))

MESSAGE_NUMBERS_UNIQUE_WITHIN_SESSION:
  all m1, m2 in JsonRpcMessage:
    (session(m1) = session(m2) and message_number(m1) = message_number(m2)) => (m1 = m2)

REQUEST_IDS_UNIQUE_PER_SENDER_AND_SESSION:
  all r1, r2 in Request:
    (session(r1) = session(r2) and sender(r1) = sender(r2) and request_id(r1) = request_id(r2)) => (r1 = r2)

RESPONSES_REVERSE_REQUEST_DIRECTION:
  all r in ReplyMessage:
    session(r) = session(response_to(r))
    sender(r) = receiver(response_to(r))
    receiver(r) = sender(response_to(r))

-- Lifecycle invariants:
HTTP_SESSIONS_USE_STREAMABLE_HTTP:
  all s in HttpSession:
    session_transport(s) = StreamableHttpTransport

INITIALIZE_REQUEST_DIRECTION:
  all r in InitializeRequest:
    sender(r) = session_client(session(r))
    receiver(r) = session_server(session(r))

INITIALIZE_IS_FIRST_MESSAGE:
  all s in Session:
    all m in JsonRpcMessage:
      (session(m) = s and not (m = initialize_request(s))) => (message_number(initialize_request(s)) < message_number(m))

INITIALIZE_RESULT_REPLIES_TO_INITIALIZE_REQUEST:
  all s in Session:
    response_to(initialize_result(s)) = initialize_request(s)
    session_protocol_version(s) = negotiated_protocol_version(initialize_result(s))

INITIALIZED_NOTIFICATION_DIRECTION:
  all n in InitializedNotification:
    sender(n) = session_client(session(n))
    receiver(n) = session_server(session(n))

INITIALIZED_FOLLOWS_INITIALIZE_RESULT:
  all s in Session:
    message_number(initialize_result(s)) < message_number(initialized_notification(s))

    sender(n) = sender(cancellation_request(n))
    receiver(n) = receiver(cancellation_request(n))
